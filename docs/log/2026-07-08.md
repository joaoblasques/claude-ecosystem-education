## Extended Thinking (step 3)

First tried it on a health-data incremental-ingestion design (late-arriving clinical
events with corrections) — and immediately learned something the exercise didn't ask
for: I *couldn't evaluate either answer*, because the question sat at the edge of what I
know cold. The thinking version "looked more complete," but I had to admit that
**"looks more complete" is a presentation signal, not a correctness one.** You can't
grade an answer you can't follow.

So I re-ran the same question as a topic I *do* know cold, and then I could actually
judge both:

- The thinking version's real edge wasn't length — it was **finding the load-bearing
  frame**: "keep two clocks separate, never watermark on the wrong one" (event_time vs
  change_time). The one-shot answer led with a mechanism (merge + watermark) and buried
  that idea; the thinking answer made it the organizing principle.
- It also caught **two gaps the one-shot missed**: late-arriving *dimensions* (a fact
  landing before its patient/clinician dimension), and **propagating corrections to
  consumers** who may have already acted on the stale value ("pipeline correctness ≠
  clinical correctness").
- But it wasn't a clean sweep: the **one-shot answer caught retractions / soft-deletes**
  (entered-in-error events, tombstones via CDF) that the thinking version dropped.

**Verdict:** Extended Thinking earns its latency on multi-constraint design reasoning —
it *reorganizes* an answer around the real crux rather than just padding it — but it is
**not strictly dominant**. Read both critically. And the deeper lesson is about me, not
the toggle: judge outputs on content you can evaluate, and when you can't, that's the
signal to close the knowledge gap first (Day 6's whole point, arriving early).

# 2026-07-09 — Day 2 (Level 1, Day 2): Artifacts

**Shipped:** a job-application tracker artifact — built from a chat inside my
`Level 1 — Claude Apps` project, which produced a Cowork working folder with a real
`job-tracker.html` file. Started from the minimal version (table + Add-row button) to
iterate up from there.

## What I did

- Learned the Artifacts create flow the hard way: the sidebar "Artifacts → create new"
  offers "chat or cowork," not a project binding — so the clean route is *start a chat
  inside the project and just ask for the artifact*.
- Built the minimal tracker first (columns + one button) rather than specifying
  everything up front — the mini-version-first rule.

## What broke / surprised me

- The big one: **"artifact" means two different things.** An in-chat artifact is a live
  preview that lives in the message thread; what I got was a **Cowork artifact** — Claude
  opened a working folder and wrote an actual `job-tracker.html` *file* to disk. The URL
  even says `cowork-artifact`. So "I built it in my project but it's a file in a folder"
  is expected, not a bug — Cowork launched from inside the project chat and produced a
  durable file. (This was under-explained on the Day-2 page; I fixed the curriculum page
  to carry the preview-vs-file distinction.)
- Useful mental model that fell out: **Projects organize chats; Cowork is a separate
  workspace that writes files; an artifact's "home" depends on which surface built it.**

## Iteration — Day 2 closed

Ran the one-feature-at-a-time loop on the tracker: status dropdown → color-by-status →
single-status filter, each as its own message. **All three worked first try, no
surprises** — nothing over-built, nothing I didn't ask for. That's the actual lesson
landing: each scoped change succeeds cleanly, where cramming all three into the original
prompt would've produced something half-broken to debug. Mini-version-first + one change
per message isn't ceremony — it's what makes artifacts reliable.
